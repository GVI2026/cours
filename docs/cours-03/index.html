Cours 03

Flows de développement — suite, conclusion et transition vers la CI

Cette séance termine le module sur les flows commencés au cours-02, puis montre pourquoi ces flows amènent naturellement à parler de DevOps, d'intégration continue et de pipelines. Vous allez comparer plusieurs stratégies de branches, puis manipuler une vraie CI sur un dépôt NestJS simple.

  • Comprendre quand GitLab Flow devient pertinent face à GitHub Flow ou Git Flow.
  • Relier les flows de développement aux environnements et au besoin d'automatisation.
  • Stabiliser le vocabulaire minimal autour de DevOps, CI, Delivery, Deployment et pipeline.
  • Diagnostiquer et corriger une chaîne CI cassée à partir de logs et d'un workflow GitHub Actions réel.

QCM de reprise du cours-02

La séance commence par 20 minutes de QCM sur papier. Les questions portent sur les notions du cours-02 : Docker, GitHub Flow, Git Flow, Pull Requests, release branches, hotfixes et logique globale des workflows étudiés.

Une fois le QCM terminé, nous repartons du point où le cours-02 s'arrêtait : que se passe-t-il quand un projet doit maintenir plusieurs versions actives et quand la coordination Git devient plus coûteuse ?

GitLab Flow

GitHub Flow fonctionne très bien tant qu'on intègre en continu sur une seule ligne courante. Mais certains produits doivent aussi continuer à corriger une version plus ancienne déjà déployée chez des clients. C'est là qu'un flow de type GitLab Flow devient intéressant.

Deux lignes actives, deux contraintes différentes

Dans le dépôt de démonstration, main représente la ligne courante du produit, alors que production/v1 représente une version plus ancienne encore supportée. Une nouvelle fonctionnalité part de main. Un correctif urgent pour les utilisateurs de la v1 part de production/v1.

Lien rapide avec les environnements

Ce type de flow fait rapidement apparaître des contextes de diffusion différents : environnement de développement pour la ligne courante, environnement de validation intermédiaire, puis production pour une version donnée. Nous resterons aujourd'hui sur trois repères utiles : DEV, STAGING et PROD.

Le coût caché : coordonner les correctifs

Dès qu'un bug existe à la fois sur la version maintenue et sur la ligne courante, il faut le reporter aussi sur main après correction sur production/v1. Cela peut se faire par cherry-pick ou par correction équivalente, mais dans tous les cas cela augmente le coût de coordination.

Si vous devez ajouter une nouvelle fonctionnalité, quelle branche cible devez-vous choisir ?

Vous devez partir de main, car une fonctionnalité nouvelle concerne la ligne courante du produit.

Une branche de maintenance comme production/v1 sert d'abord à corriger une version plus ancienne déjà utilisée, pas à enrichir la version courante.

Mini-démo guidée — dépôt GitLab Flow

Dépôt support : GVI2026/tp-gitlab-flow

Cette démo doit durer 10 minutes. L'objectif n'est pas de refaire un TP complet, mais de rendre visible un besoin que GitHub Flow ou Git Flow couvrent moins bien : maintenir une ancienne version tout en développant la suivante.

  1. Vérifier l'existence des deux branches permanentes : 
    git branch -a
  2. Montrer une nouvelle fonctionnalité sur la ligne courante avec l'endpoint GET /tasks/stats, à implémenter depuis feature/add-task-stats.
  3. Basculer ensuite sur la ligne de maintenance et ouvrir un hotfix depuis production/v1 pour corriger la validation du champ title. 
    git checkout production/v1
git checkout -b hotfix/fix-title-validation
  4. Finir par verbaliser le vrai message du flow : une correction critique ne reste pas enfermée sur une seule branche si le bug existe aussi sur la ligne courante.
Pourquoi cette démo prépare-t-elle bien la transition vers la CI ?

Parce qu'à partir du moment où plusieurs lignes de travail sont actives, chaque correctif ou chaque intégration doit être sécurisée plus vite et plus souvent.

Plus le flow se complexifie, plus l'automatisation devient un moyen concret de garder confiance dans les changements.

Trunk-Based Development — teaser stratégique

Le Trunk-Based Development n'est pas le sujet pratique d'aujourd'hui. Nous le regardons comme une direction possible pour des équipes qui intègrent très fréquemment et qui disposent déjà d'une CI/CD solide.

Principe : une branche principale unique, souvent main ou trunk, sur laquelle les intégrations sont très fréquentes.

Feature flags : ils permettent de masquer une fonctionnalité incomplète sans la garder éternellement sur une branche longue.

Merge queues : elles sérialisent les intégrations pour réduire le risque de casser la branche principale au moment du merge.

Point clé : sans pipeline fiable, rapide et lisible, ce mode de travail devient très difficile à tenir.

Pourquoi le Trunk-Based Development est-il présenté comme la suite du module, et non comme un flow à pratiquer tout de suite ?

Parce qu'il suppose déjà une maturité CI/CD importante : tests rapides, build fiable, feedback court et forte discipline d'intégration.

Les prochains cours sur la CI/CD sont justement ce qui permettra de comprendre comment atteindre cette maturité.

Monorepo vs polyrepo — comparaison brève

Cette comparaison n'est pas un chapitre autonome. Elle sert seulement à enrichir la conclusion : selon le contexte, la gestion des versions, des permissions et des pipelines peut devenir plus ou moins simple.

Monorepo

Tout le code vit dans un seul dépôt. On y gagne souvent en cohérence, mais on complexifie plus vite la gestion des droits, la lecture des pipelines et la question des versions compatibles entre plusieurs composants.

Polyrepo

Chaque service ou composant a son dépôt. On gagne en isolation, mais la synchronisation entre versions et la gestion des compatibilités demandent un outillage et une discipline plus visibles.

Tableau comparatif final des flows

Aucun flow n'est universel. Un projet peut très bien évoluer d'un mode très simple vers un flow plus structuré à mesure que la taille de l'équipe, le nombre de versions supportées et l'exigence qualité augmentent.

Sans flow réel : acceptable au tout début d'un prototype individuel, mais fragile.

GitHub Flow : bon point d'entrée dès qu'on veut intégrer souvent sur une ligne courante claire.

GitLab Flow : pertinent dès qu'il faut gérer plusieurs versions actives d'un même produit.

Trunk-Based Development : logique quand la maturité CI/CD permet des intégrations très fréquentes sur un tronc unique.

Cette trajectoire synthétise la progression réaliste d'un projet.

Chargement du schéma…

Quelle idée devez-vous retenir avant de passer à la CI ?

Plus un flow vise des intégrations fréquentes ou la gestion de plusieurs versions, plus l'automatisation devient indispensable.

La CI n'arrive donc pas comme un nouveau sujet déconnecté : elle répond à une difficulté déjà visible dans les flows.

Dépôt CI pédagogique — temps 1 : prise en main guidée

Dépôt support : GVI2026/tp-ci-api

Ce premier temps dure 20 minutes. Le but n'est pas encore de corriger la pipeline, mais de stabiliser l'environnement de travail et de rendre le dépôt familier avant le cours sur la CI/CD.

  1. Cloner le dépôt puis l'ouvrir dans VS Code. Si le DevContainer est proposé, l'accepter pour homogénéiser l'environnement.
  2. Lancer l'application : 
    npm run start:dev
    Vérifier ensuite que Swagger répond sur http://localhost:3000/api.
  3. Repérer trois fichiers utiles pour la suite : .github/workflows/ci.yml, package.json et le dossier src/.
  4. Faire une première lecture du workflow pour identifier ce qui sera discuté ensuite : triggers, jobs, steps, ordre des vérifications.
Pourquoi insiste-t-on sur la prise en main avant la théorie ?

Parce qu'un dépôt inconnu rend le vocabulaire CI abstrait. Si vous avez déjà vu l'application, le workflow et l'arborescence, les notions théoriques deviennent beaucoup plus concrètes.

Pont vers DevOps et CI/CD

Le pont logique est le suivant : plus une équipe livre souvent ou maintient plusieurs lignes en parallèle, plus elle doit détecter tôt les problèmes. DevOps et la CI/CD cherchent précisément à réduire ce délai entre un changement et la confiance qu'on peut lui accorder.

DevOps

L'idée n'est pas de fusionner tous les métiers, mais de mieux relier développement, livraison et exploitation. On s'intéresse donc à la vitesse de mise à disposition, à la stabilité et à la capacité à restaurer rapidement un service quand un problème survient.

Les métriques DORA en lecture rapide

Deployment Frequency : à quelle cadence l'équipe déploie.

Lead Time for Changes : temps entre un changement validé et sa mise en service.

MTTR : temps moyen nécessaire pour restaurer le service après un incident.

Change Failure Rate : part des changements qui provoquent un problème en production.

Reliability : capacité à tenir les objectifs de service.

Environnements utiles aujourd'hui

DEV : environnement de travail et d'expérimentation. Dans ce cours, nous allons surtout manipuler un environnement local.

STAGING : environnement de validation proche de la production, utile avant une mise en service réelle.

PROD : environnement utilisé par les vrais utilisateurs.

SIT, UAT et PREPROD existent aussi, mais ils ne sont pas notre sujet principal aujourd'hui.

CI, Continuous Delivery, Continuous Deployment

Continuous Integration : on vérifie automatiquement qu'un changement s'intègre bien au reste du projet.

Continuous Delivery : l'application reste dans un état livrable, mais la mise en production peut rester manuelle.

Continuous Deployment : le passage en production est lui aussi automatisé.

Anatomie minimale d'une pipeline

Trigger : événement qui lance le workflow, par exemple un push ou une pull_request.

Workflow : orchestration globale décrite dans un fichier YAML.

Job : bloc logique d'exécution, par exemple formatage, tests ou build.

Step : action unitaire exécutée dans un job.

Runner : machine qui exécute les steps.

Logs : source principale de diagnostic quand une étape échoue.

Fail-fast : on place les vérifications les plus rentables tôt pour arrêter rapidement une chaîne déjà invalide.

DAG : graphe d'exécution qui rend visibles les dépendances entre jobs et les possibilités de parallélisation.

Secrets masqués : les outils de CI évitent d'afficher les valeurs sensibles dans les logs, même si leur gestion reste un sujet sérieux.

Dépôt CI pédagogique — temps 2 : exercices et correction

Le but est maintenant de diagnostiquer puis corriger une CI cassée. Vous travaillez en binôme. Le dépôt a été conçu pour que les pannes soient réalistes, lisibles et rapides à interpréter.

Pipeline de référence à garder en tête 

install → format-lint → tests → build → security

L'ordre n'est pas arbitraire. Il sert à détecter vite les erreurs les moins coûteuses à diagnostiquer avant de mobiliser des étapes plus longues.

Commandes utiles pendant le TP 

npm run format:check
npm run lint
npm run test:ci
npm run build
act
act -j format-lint
act -j tests

Exercice 1 — corriger une étape qualité simple

Branche : exercise/01-format-lint. L'objectif est de lire le bon log, puis de corriger un problème de formatage ou de lint sans modifier inutilement le reste du projet.

Exercice 2 — réparer un test unitaire

Branche : exercise/02-test-failure. L'échec est volontairement simple à comprendre : vous devez repérer le test, relire l'intention puis corriger soit le test, soit le code testé.

Exercice 3 — remettre la pipeline dans un ordre fail-fast

Branche : exercise/03-pipeline-order. Le travail consiste à réorganiser les dépendances de jobs dans .github/workflows/ci.yml pour retrouver une chaîne logique et lisible.

Exercice 4 — lire les logs et trouver la bonne cause

Branche : exercise/04-log-reading. Ici, l'enjeu n'est pas seulement de corriger : il faut surtout savoir lire le bon job, la bonne step et le bon message d'erreur.

Exercice 5 — stabiliser la chaîne complète

Branche : exercise/05-stabilize. Cette branche sert à vérifier que vous savez appliquer un ordre de diagnostic cohérent jusqu'au retour complet au vert.

Bonus — paralléliser une partie simple du DAG

Branche : bonus/parallelization. Ce bonus s'adresse aux binômes les plus rapides. Le but est de séparer proprement des vérifications indépendantes sans rendre la lecture du graphe inutilement compliquée.

Quel ordre de diagnostic devez-vous appliquer quand tout semble cassé ?

Commencez par les étapes qualité les plus précoces, puis les tests, puis le build, puis le reste. L'idée n'est pas d'ouvrir tous les fronts à la fois, mais d'éliminer les causes visibles dans l'ordre où la pipeline les rencontre.

Correction collective et débrief

  • Quels échecs étaient visibles en premier ?
  • Dans quel job fallait-il lire les logs ?
  • Pourquoi l'ordre des jobs n'est-il pas arbitraire ?
  • Qu'est-ce qu'une parallélisation utile, et qu'est-ce qui ne l'est pas ?

Synthèse et marge pédagogique

En trois séances, vous êtes passés d'un usage local de Git à des workflows d'équipe, puis à la question suivante : comment garder confiance dans les changements quand les branches, les versions et les intégrations se multiplient ?

La réponse du jour est claire : avant de parler de déploiement, il faut déjà savoir intégrer, vérifier, diagnostiquer et corriger rapidement. C'est exactement le rôle de la CI.

Les 10 dernières minutes servent de marge pédagogique : questions, retard éventuel, reprise d'un point du TP ou reformulation d'une notion de vocabulaire si nécessaire.

Pour la suite du module, gardez cette idée directrice : plus on veut intégrer souvent et livrer vite, plus il faut raccourcir le temps entre un changement et le retour fiable que donne la pipeline.
